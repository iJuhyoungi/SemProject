# NOR flash Status 레지스터 — WIP 과 WEL 이 지배하는 것

> 대상 칩: **ISSI IS25LP064** (JEDEC ID `9D 60 17`, 8MB). F-2 에서 직접 읽어 확정했다.
> 요약: flash 에 쓰기를 시도하기 전에 반드시 알아야 할 8개 비트. 이 중 **WIP**(바쁨)과
> **WEL**(허락됨) 두 개가 "명령이 실행될지 조용히 무시될지"를 결정한다.

---

## Part 1. 비트 지도 — 8비트 안에 뭐가 들어있나

Status 레지스터는 `0x05` 명령 한 번으로 읽는 1바이트다. 작지만 flash 의 현재 상태를
전부 담고 있다.

```
 bit7    bit6    bit5   bit4   bit3   bit2    bit1    bit0
┌──────┬──────┬──────┬──────┬──────┬──────┬───────┬───────┐
│ SRWD │  QE  │ BP3  │ BP2  │ BP1  │ BP0  │  WEL  │  WIP  │
└──────┴──────┴──────┴──────┴──────┴──────┴───────┴───────┘
   0      1      0      0      0      0    0 또는 1    0
   └──────────── 우리 보드의 실제 값 ────────────────────┘
```

| 비트 | 이름 | 뜻 | 우리 값 |
|---|---|---|---|
| 0 | **WIP** | Write In Progress — flash 가 내부 작업 중 | 0 (한가함) |
| 1 | **WEL** | Write Enable Latch — 쓰기 허락됨 | 상황따라 |
| 2~5 | BP0~BP3 | Block Protect — 보호 영역 지정 | 0000 (보호 없음) |
| 6 | **QE** | Quad Enable — 4가닥 모드 허용 | **1 (건드리면 안 됨)** |
| 7 | SRWD | Status Register Write Disable (WP# 핀 연동) | 0 |

> **주의**: QE 비트 위치는 제조사마다 다르다. ISSI 는 **SR1 의 bit6** 이지만
> Winbond 계열은 **SR2 의 bit1** 이다. 칩을 바꾸면 이 지도부터 다시 그려야 한다.

### 우리가 실제로 보는 두 값

```
0x42 = 0100 0010     0x40 = 0100 0000
       ↑      ↑             ↑
      QE=1  WEL=1          QE=1, WEL=0
```

두 값의 차이는 **오직 bit1(WEL) 하나**다. `0x40` 이 평상시 바탕이고, 여기에
`0x02`(WEL)가 얹히면 `0x42` 가 된다.

`0x40`(QE)이 항상 켜져 있는 이유는 **부팅 FCB 가 세운 것**이다. 우리 XIP 는
`0xEB` Quad I/O Read 로 4가닥 읽기를 하는데, QE=0 이면 flash 가 4가닥 모드를 거부한다.
즉 **QE 를 실수로 끄는 순간 코드 fetch 가 죽는다.** status 를 다룰 때 이 비트는
"읽기만 하고 절대 안 건드리는" 대상이다.

## Part 2. WIP — "지금 바쁘다"

### 어떻게 변하나

**하드웨어가 자동으로** 세우고 내린다. 소프트웨어가 직접 쓸 수 없다.

```
erase/program 명령 접수   →  WIP: 0 → 1   (자동)
내부 고전압 작업 완료      →  WIP: 1 → 0   (자동)
```

### WIP 값에 따라 flash 가 어떻게 달라지나

| WIP | flash 상태 | 받아들이는 명령 | 거부하는 명령 |
|---|---|---|---|
| **0** | 한가함 | **전부** | 없음 |
| **1** | 내부 작업 중 | `0x05`(status), `0x9F`(ID) 정도만 | **읽기·쓰기·지우기 전부 무시** |

여기가 핵심이다. **WIP=1 인 동안 flash 는 데이터 읽기 요청에 응답하지 않는다.**
이것이 두 가지 결과를 낳는다.

**① 반드시 폴링해야 한다.** erase 명령을 쏘고 바로 다음 명령을 보내면 그냥 씹힌다.
"지웠는데 왜 안 써지지?"의 대부분이 이것이다. `0x05` 로 WIP 이 0 이 될 때까지 계속
물어봐야 한다 — status 읽기는 바쁜 중에도 되는 유일한 창구니까.

**② XIP 가 죽는다.** CPU 는 코드를 flash 에서 fetch 한다. 그런데 erase 중에는 flash 가
읽기를 안 준다. **erase 명령을 쏜 직후의 코드가 flash 에 있으면, 다음 명령어를 못 가져와
즉사한다.** 이것이 erase 구간을 **ITCM RAM 함수**로 올려야 하는 이유다.

### 시간 감각

| 작업 | WIP=1 지속 시간 |
|---|---|
| Page program (256B) | 1ms 미만 |
| **Sector erase (4KB)** | **수십 ms** |
| Chip erase (8MB) | 수십 초 |

수십 ms 는 CPU 에게 **수백만 사이클**이다. 그래서 MCAL Fls 가 블로킹이 아닌
**비동기 job 모델**(요청 → `Fls_MainFunction` 폴링 → `MEMIF_JOB_PENDING`)인 것이다.

## Part 3. WEL — "쓰기 허락받음"

### 어떻게 변하나

이것도 직접 못 쓴다. **전용 명령으로만** 조작된다.

```
WREN (0x06)                       →  WEL: 0 → 1
WRDI (0x04)                       →  WEL: 1 → 0
쓰기 명령이 완료됨(erase/program)  →  WEL: 1 → 0   (자동! 한 번 쓰면 스스로 잠긴다)
```

마지막 줄이 중요하다. **한 번 쓰면 자동으로 다시 잠긴다.** erase 를 두 번 하려면
WREN 도 두 번 보내야 한다.

### WEL 값에 따라 flash 가 어떻게 달라지나

| WEL | 파괴적 명령(erase/program)을 쏘면 |
|---|---|
| **0** | **조용히 무시.** 에러도, 응답도 없음 — 아무 일도 안 일어남 |
| **1** | 실행됨 |

"조용히 무시"가 악질이다. **에러가 안 난다.** `IPCMDDONE` 은 정상적으로 뜨고 `INTR` 에
에러 비트도 안 선다. 드라이버 입장에선 "명령 잘 나갔다"인데 flash 는 아무것도 안 했다.
그래서 **erase 후 검증(읽어서 `0xFF` 확인)이 필수**다.

### 왜 이렇게 번거롭게 만들었나 — 이중 방아쇠

폭주한 코드나 노이즈가 우연히 `0x20`(sector erase) 한 바이트를 SPI 라인에 흘렸다고 하자.
WEL 이 없다면 그 순간 4KB 가 날아간다. WEL 이 있으면 flash 는 **허락받지 않은 명령으로
보고 조용히 무시**한다. 서로 다른 두 명령이 우연히 올바른 순서로 나갈 확률은 거의 없다.

**워치독의 매직 시퀀스(`0xC520`→`0xD928`)와 똑같은 철학** — 중요한 동작일수록 "실수로는
못 하게" 만든 것이다. 그리고 한 번 쓰면 자동으로 잠기니 연속 erase 사고도 막힌다.

## Part 4. WIP × WEL 조합표 — 실전 판정

| WIP | WEL | 상태 | erase/program 을 쏘면 | 언제 보이나 |
|---|---|---|---|---|
| 0 | 0 | 한가하고 **잠김** | ❌ 무시됨 | 평상시 (`SR=0x40`) — **WREN 빠뜨린 실수** |
| 0 | 1 | 한가하고 **허락됨** | ✅ **실행됨** | WREN 직후 (`SR=0x42`) — 유일하게 동작하는 조합 |
| 1 | 0/1 | **작업 중** | ❌ 무시됨 | erase/program 진행 중 |

기억할 문장은 하나다 — **`WIP=0 && WEL=1` 일 때만 쓰기가 나간다.**

### 세 번째 관문 — BP0~BP3

우리 보드는 BP 가 전부 0(보호 없음)이라 문제가 없지만, 이 비트가 세워져 있으면
**WEL=1 이어도 해당 영역의 erase/program 이 거부된다.** "WREN 도 했고 WIP 도 0 인데
안 지워진다"면 BP 비트를 의심 목록에 넣어야 한다.

## Part 5. 실측 기록 (F-4a, 2026-07-21)

WREN/WRDI 로 래치를 직접 조작하며 관찰한 결과. flash 내용은 전혀 바꾸지 않는 안전한
실험이다.

```
[FLS] --- WEL latch test (flash 내용 변경 없음) ---
[FLS]   boot   : SR=0x42 WIP=0 WEL=1
[FLS]   WRDI   : SR=0x40 WIP=0 WEL=0     ← 래치 내려감
[FLS]   WREN   : SR=0x42 WIP=0 WEL=1     ← 래치 올라감
[FLS]   relock : SR=0x40 WIP=0 WEL=0     ← 다시 잠김
```

**부팅 직후 WEL 이 이미 1 이었던 것**은 부팅 ROM 이 QE 비트를 세우려고 status write 를
하면서 남긴 잔재였다. WRDI 로 내려가고 WREN 으로 올라가는 것이 확인되어, 래치 자체는
정상 동작함이 증명됐다. `0x40`(QE)은 네 번 모두 유지됐다 — 우리 실험이 XIP 를 위협하지
않았다는 뜻이다.

## Part 6. 쓰기 명령의 정석 흐름

지금까지의 내용이 실제 erase 에서 이렇게 엮인다.

```
① WREN (0x06)             → WEL: 0→1
② status 읽어 WEL=1 확인    ← 안 서 있으면 다음 단계가 무의미
③ Sector Erase (0x20)      → 내부 작업 시작, WIP: 0→1
④ WIP 폴링 (0x05 반복)      → 수십 ms 동안 WIP=1, 이 구간은 ★RAM 함수★
⑤ WIP=0 → 완료             → WEL 도 자동으로 0 (다시 잠김)
⑥ IP read 로 0xFF 확인      → 진짜 지워졌는지 검증
```

②와 ⑥이 **검증 이중 안전장치**다. WEL 은 조용히 안 서고 erase 는 조용히 무시되니,
앞뒤로 눈으로 확인하는 것 외에는 방법이 없다.
