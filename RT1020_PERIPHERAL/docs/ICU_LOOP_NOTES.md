# Icu 확장 — PWM 트리거를 XBAR 로 되받아 측정하는 내부 루프백

> 참조: RM Rev.2 — Table 4-5/4-7 (XBAR 배정표), Ch.56 (XBARA), Ch.50 (FlexPWM TCTRL), Ch.11 (IOMUXC_GPR)
> 요약: PWM 이 내보내는 트리거 신호를 칩 내부에서 QuadTimer 입력으로 연결한다.
> **Pwm 에 설정한 주기를 Icu 가 실제로 측정**하게 되므로, 두 드라이버가 서로의 동작을 확인해 준다.
> 부품, 점퍼, 계측기 없이 가능하다.

---

## Part 1. 배경 — 왜 이걸 만드는가

### 지금까지의 Pwm 검증은 반쪽이었다

P-8 에서 Pwm 을 검증할 때는 SM0CNT 카운터가 증가하는지, VAL3 에 쓴 값이 제대로 읽히는지를
확인했다. 둘 다 **PWM 모듈의 레지스터를 읽어 본 것**일 뿐이다. 카운터가 돌고 있어도 출력 신호가
설정한 주기대로 나오고 있는지는 별개 문제다. 신호를 직접 보면 되지만, EVK 에서 FlexPWM 출력 핀은
SDRAM 라인에 배정되어 있어 스코프 없이는 볼 방법이 없었다.

### CAN 검증에서 썼던 발상을 다시 쓴다 — loopback

CAN 을 검증할 때는 트랜시버 없이 loopback 모드로 TX 신호를 RX 로 되돌려 받았다. 보낸 데이터가
그대로 수신되는지 확인하는, 출력을 입력으로 되받는 방식이었다. PWM 에도 같은 발상을 적용할 수
있다. 다만 CAN 은 컨트롤러에 loopback 기능이 내장되어 있었고, PWM 과 타이머 사이에는 그런
기능이 없다. 대신 i.MX RT 에는 **XBAR(crossbar switch)** 가 있다.

XBAR 는 페리페럴들의 출력 신호(입력 88개)와 페리페럴들의 입력(출력 100개 이상)을
**레지스터 설정만으로 연결해 주는 신호 스위치**다. 이걸 쓰면 보드에 선 하나 대지 않고
다음 경로가 만들어진다.

```
FlexPWM SM0 ──── 주기가 한 바퀴 돌 때마다 트리거 펄스 1개 (VAL1 비교 이벤트)
     │
   XBAR1 ─────── 입력 40번을 출력 88번으로 연결 (레지스터 설정 한 번)
     │
QuadTimer1 ch2 ── 펄스 개수를 카운트 (cascade 데모가 쓰는 ch0/ch1 은 그대로 둠)
```

이 경로가 열리면 **Pwm 설정값이 Icu 측정값으로 되돌아온다.** Cfg.c 에서 `Pwm_Config.period`
를 절반으로 줄였을 때 Icu 가 세는 펄스 수가 두 배가 되는 것까지 확인하면, Config 패턴의 실증과
드라이버 상호 검증을 한 실험으로 끝낼 수 있다.

## Part 2. 신호 경로 (RM 에서 확정한 사실)

| # | 구간 | 레지스터 | 값/비트 | 근거 |
|---|---|---|---|---|
| ① | 트리거 출력 켜기 | `SM0TCTRL` (FLEXPWM1 + **0x2A**) | `OUT_TRIG_EN` bit1 — VAL1 비교 시 트리거 발생 | Ch.50.8.22 (SM0=0x2A, SM1=0x8A ...) |
| ② | XBAR 입력 | — | **XBAR1_IN40** = SM0 의 OUT_TRIG0 **OR** TRIG1 | Table 4-5 |
| ③ | XBAR 클럭 | `CCM_CCGR2` | **CG11** (bits 23:22) = xbar1 게이트 | Ch.14 CCGR2 |
| ④ | XBAR 연결 | `XBARA1_SEL44` (**0x403B_C000 + 0x58**) | bits **[6:0] = 40** → 출력 88번의 소스를 입력 40번으로 | Ch.56.5.45 |
| ⑤ | XBAR 출력 | — | **XBAR1_OUT88** = QTIMER1_TIMER2 입력 | Table 4-7 |
| ⑥ | QTimer 입력 선택 | `IOMUXC_GPR6` (0x400A_C000 + **0x18**) | bit **2** = 1 → TIMER2 입력을 pad 대신 XBAR 로 | Ch.11 GPR6 |
| ⑦ | 펄스 카운트 | `TMR1_CTRL(2)` | CM=001 (상승엣지 카운트), **PCS=0010** (counter 2 입력) | Ch.49 — P-12 와 같은 패턴, 소스만 외부 입력 |

정리하면: **TCTRL 로 트리거를 내보내고, CCGR2·SEL44·GPR6 으로 경로를 연결하고, ch2 로 센다.**

세부 사항 몇 가지.

- SELn 레지스터는 16-bit 하나에 출력 두 개 분량의 선택값이 들어간다. 짝수 번호 출력은 [6:0],
  홀수 번호 출력은 [14:8] 이다. 출력 88번은 짝수이므로 SEL44 의 하위 필드에 40 을 쓴다.
- VAL1 은 홀수 번호 레지스터라서 트리거가 OUT_TRIG1 쪽으로 나온다. 그런데 XBAR1_IN40 은
  TRIG0 과 TRIG1 을 OR 로 묶어 받으므로 어느 쪽으로 나오든 문제없다.
- 이 설계가 성립하는 열쇠는 ⑥의 GPR6 스위치였다. QuadTimer 입력이 물리 핀으로만 연결되는
  구조였다면 점퍼 없이는 불가능했을 텐데, RT1020 은 입력을 pad 와 XBAR 중에서 고를 수 있다.

### 무엇을 측정하는 것인가 — 파형이 아니라 이벤트 펄스

OUT_TRIG 로 나오는 신호는 PWM 파형 그 자체가 아니라 **VAL 레지스터 비교 이벤트를 알리는
펄스**다. VAL1(주기의 끝) 트리거만 켜 두면 PWM 이 한 주기를 돌 때마다 펄스가 정확히 1개씩
나온다. 그래서 이번에 측정하는 것은 주기(주파수)다. duty 까지 재려면 VAL2/VAL3 트리거를
별도 XBAR 경로로 빼서 캡처 타임스탬프 두 개를 비교해야 하므로, 다음 과제로 남겨 둔다.

### 예상 수치와 합격 기준

```
FlexPWM 클럭 = ipg (tick 역산 때 확인했듯 60MHz 근처), PRSC = ÷1
PWM 주기 = 0xFFFF tick  →  트리거 주파수는 대략 1kHz 수준
beat(약 0.29초) 하나 동안 수백 개가 세어질 것으로 예상
```

ipg 의 정확한 값을 모르는 상태이므로 합격 기준은 절대값이 아니라 **비율**로 잡는다.
Cfg 에서 period 를 절반으로 바꿨을 때 측정되는 펄스 수가 2.0배가 되면 경로 전체가 정상이라는
뜻이다. 확실하지 않은 상수가 남아 있을 때는 비율로 판단하는 쪽이 안전하다 — GPT tick 주기를
역산으로 바로잡았던 경험에서 가져온 원칙이다.

## Part 3. 설계 — 어느 코드를 어느 층에 두는가

계층을 나누는 기준이 이번에도 그대로 적용된다.

| 작업 | 위치 | 이유 |
|---|---|---|
| XBAR 연결 + GPR6 (③④⑥) | **Port.c** — `Port_RoutePwmTrigToQtmr()` | XBAR 는 페리페럴 사이의 신호 라우팅이다. 핀과 신호의 경로를 정하는 일은 Port 드라이버 몫이고, 실무 MCAL 에서도 XBAR 설정은 Port 가 맡는다 |
| 트리거 출력 켜기 (①) | **Pwm.c** register 층 — `Pwm1_Init` 에 한 줄 | PWM 모듈 자신의 레지스터이기 때문 |
| ch2 카운터 (⑦) | **Icu.c** register 층 — `Icu1_Init` 확장 | QuadTimer 모듈 자신의 레지스터이기 때문 |
| 측정값 읽기 | **Icu facade** — `Icu_GetEdgeNumbers(1)` | 채널 0 = cascade(기존), 채널 1 = PWM 측정(신규). guard 조건을 `Channel > 1` 로 완화 |

rt1020_regs.h 에 새로 추가할 매크로는 `XBARA1_SEL44`, `IOMUXC_GPR_GPR6`, `FLEXPWM1_SM0TCTRL`
세 개다. TMR1 ch2 는 기존 채널 매크로(`TMR1_CTRL(n)` 등)를 그대로 쓴다.

main 데모에는 기존 `[Icu] edge count`(cascade) 옆에 `[Icu] pwm edges` 한 줄을 추가한다.
beat 마다 일정한 양씩 늘어나면 경로가 살아 있다는 뜻이다.

## Part 4. 검증 계획 (A2-c)

```
① 기준선 — pwm edges 가 beat 마다 일정하게 증가하는지 확인 + 기존 데모 전체 회귀
② 루프백 검증 ⭐ — Cfg.c 의 Pwm_Config.period 를 0xFFFF → 0x7FFF (절반) 로 변경
   → 드라이버 코드는 한 줄도 바꾸지 않았는데 pwm edges 증가율이 2.0배가 되어야 함
   → cascade 채널은 PWM 과 무관하므로 변화가 없어야 함 — 대조군 역할
   → duty 환산식이 config 의 period 를 참조하므로 VAL3 도 0x1FFF 로 함께 바뀜 (Config 실증 재확인)
③ 원복 후 기준선으로 돌아오는지 확인 → 커밋
```

②에서 cascade 채널이 대조군 역할을 한다는 점이 이 실험의 좋은 부분이다. pwm edges 만 두 배가
되고 cascade 쪽은 그대로라면, 변화가 PWM → XBAR → ch2 경로에서 왔다는 사실이 그 자체로
증명된다.
