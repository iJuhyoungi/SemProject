# 📅 DevLog: i.MX RT1020 베어메탈 환경 구축 및 디버깅 파이프라인 완성

## 🎯 1. 칩 부팅 및 메모리 아키텍처 설계
운영체제(OS)가 없는 텅 빈 칩에 전원이 들어왔을 때, 칩이 기계어를 정상적으로 읽고 `main()` 함수에 도달하기 위한 기초 공사를 진행했습니다.

* **메모리 맵 (Memory Map) & 링커 스크립트 (`.ld`)**
    * 칩의 메모리 주소 지도에 맞춰 코드(`.text`)는 Flash(`0x60000000`)에, 변수(`.data`, `.bss`)는 고속 RAM(`0x20000000`)에 배치되도록 링커 스크립트를 작성했습니다.
* **NXP 전용 부트 시퀀스 (FCB & IVT)**
    * **FCB (Flash Configuration Block):** 칩 내부의 공장 프로그램(Boot ROM)이 외부 Flash 메모리와 통신할 수 있도록 알려주는 512바이트 설정표. (오프셋 1바이트 오류도 허용하지 않는 엄격한 규격을 C 구조체로 완벽히 맞춤)
    * **IVT (Image Vector Table):** Boot ROM이 펌웨어의 진짜 진입점을 찾을 수 있도록 돕는 이정표.
* **스타트업 코드 (Startup Code) & 벡터 테이블**
    * 하드웨어 소수점 연산기(**FPU**) 활성화 및 인터럽트 주소 테이블(**VTOR**)을 재배치했습니다.
    * ISO C 표준(`-Wpedantic`)을 준수하기 위해 함수 포인터 배열 대신 `union`을 사용하여 깔끔하게 인터럽트 벡터 테이블을 구성했습니다.

> 💡 **[부가 개념] XIP (eXecute In Place)**
> i.MX RT 시리즈는 칩 내부에 코드를 저장할 공간이 없습니다. 대신 코드를 외부 Flash 메모리에 두고, 칩이 RAM으로 코드를 복사하지 않은 채 **외부 Flash에서 직접 기계어를 한 줄씩 읽어와 실행**합니다. 이를 XIP라고 부르며, FCB 설정이 조금만 틀어져도 먹통이 되는 이유가 바로 XIP 환경을 구성하지 못했기 때문입니다.

## 🛠️ 2. 하드웨어 레지스터 직접 제어 (GPIO)
데이터시트(Reference Manual)를 직접 해독하여, 라이브러리 없이 메모리 주소에 값을 써서 하드웨어를 제어하는 원리를 터득했습니다.

* **Base Address + Offset 계산법:** 3000페이지가 넘는 매뉴얼에서 모듈의 시작 주소(Base)와 특정 기능의 거리(Offset)를 더해 절대 주소를 찾아내는 훈련을 진행했습니다. (`CCM_CCGR2`, `IOMUXC`, `GPIO_DR` 등)
* **모듈 제어 3단계:** 외부 핀(LED)을 제어하기 위해 1) **CCM**으로 클럭(심장 박동)을 공급하고, 2) **IOMUXC**로 핀의 역할을 GPIO로 명시한 뒤, 3) **GPIO_GDIR/DR**로 출력 방향과 값을 설정했습니다.

> 💡 **[부가 개념] Memory-Mapped I/O (MMIO)와 `volatile` 키워드**
> CPU 입장에서는 RAM에 변수를 쓰는 것이나, 하드웨어 핀에 전기를 내보내는 것이나 **'특정 주소에 값을 쓰는 행위'**로 완전히 동일합니다. 이를 MMIO라고 합니다. 이때, 컴파일러가 "아무도 안 읽는 변수네?" 하고 코드를 삭제하는(최적화) 것을 막기 위해 하드웨어 레지스터를 다룰 때는 반드시 `volatile` 키워드를 포인터에 붙여야 합니다.

## 🚀 3. 디버깅 및 플래싱 파이프라인 구축
작성한 코드를 실제 보드(`PIMXRT1021DAG5A`)에 굽고 통신하기 위한 툴체인(OpenOCD, PyOCD, GDB)을 세팅했습니다.

* **툴체인 트러블슈팅:**
    * 공식 OpenOCD에 NXP 외부 Flash 드라이버가 없다는 한계를 파악했습니다.
    * 파이썬 기반의 `PyOCD`를 시도하며 리눅스 `PEP 668` 환경 제약(`pipx`로 해결)을 경험했습니다.
* **커스텀 OpenOCD 스크립트 수정 (`imxrt.cfg`):**
    * 서드파티 스크립트에 있던 **Reset Vector 추출 버그**를 찾아냈습니다. (ARM 표준 벡터 테이블 동작 방식과 NXP Boot ROM의 IVT 점프 방식을 혼동하여 메모리를 두 번 읽던 오류 수정)

> 💡 **[부가 개념] openocd.cfg의 마법 (Default Configuration)**
> 터미널에서 OpenOCD를 실행할 때 아무런 `-f` (설정 파일 지정) 인자를 주지 않으면, OpenOCD는 **자동으로 현재 실행 위치에서 `openocd.cfg`라는 이름의 파일을 찾아 실행**하는 기본 동작 방식(Default Behavior)을 가집니다. 이를 활용하면 긴 명령어 없이 `$OCD` 단축 명령어 하나로 서버 통신을 열어낼 수 있습니다.

> 💡 **[부가 개념] `[find ...]` 명령어와 검색 경로(Search Path)의 비밀**
> `openocd.cfg` 내부에 작성한 `source [find interface/cmsis-dap.cfg]`에서 `find`는 리눅스 명령어가 아니라 OpenOCD 내부(TCL) 함수입니다.
> OpenOCD는 실행될 때 똑똑하게도 **1) 현재 폴더(`.`)**, **2) 사용자 홈(`~/.openocd`)**, **3) 자신이 설치된 `scripts` 경로**를 '자동 검색 리스트'에 등록해 둡니다. 따라서 절대 경로를 적지 않아도, OpenOCD가 알아서 자신의 설치 폴더 깊숙한 곳에 있는 타겟/인터페이스 스크립트 파일을 귀신같이 찾아내어 읽어올 수 있습니다.

> 💡 **[부가 개념] GDB 디버깅 아키텍처**
> 디버깅은 3개의 프로그램이 협력합니다.
> 1. **GDB Client (`arm-none-eabi-gdb`):** 사람의 명령(break, run 등)을 받는 터미널.
> 2. **GDB Server (`OpenOCD`):** 그 명령을 하드웨어 전기 신호(SWD/JTAG)로 번역하는 번역기.
> 3. **하드웨어 디버거 (DAP-Link):** 실제 USB와 보드를 잇는 물리 장치.
> 코어가 1개(Single-core)인 i.MX RT1020은 GDB Server가 3333번 포트 하나만 엽니다.
